# P1089 [NOIP 2004 提高组] 津津的储蓄计划

## 题目描述

津津的零花钱一直都是自己管理。每个月的月初妈妈给津津 $300$ 元钱，津津会预算这个月的花销，并且总能做到实际花销和预算的相同。

为了让津津学习如何储蓄，妈妈提出，津津可以随时把整百的钱存在她那里，到了年末她会加上 $20\%$ 还给津津。因此津津制定了一个储蓄计划：每个月的月初，在得到妈妈给的零花钱后，如果她预计到这个月的月末手中还会有多于 $100$ 元或恰好 $100$ 元，她就会把整百的钱存在妈妈那里，剩余的钱留在自己手中。


例如 $11$月初津津手中还有 $83$ 元，妈妈给了津津 $300$ 元。津津预计$11$月的花销是 $180$ 元，那么她就会在妈妈那里存 $200$ 元，自己留下 $183$ 元。到了 $11$ 月月末，津津手中会剩下 $3$ 元钱。


津津发现这个储蓄计划的主要风险是，存在妈妈那里的钱在年末之前不能取出。有可能在某个月的月初，津津手中的钱加上这个月妈妈给的钱，不够这个月的原定预算。如果出现这种情况，津津将不得不在这个月省吃俭用，压缩预算。


现在请你根据 $2004$ 年 $1$ 月到 $12$ 月每个月津津的预算，判断会不会出现这种情况。如果不会，计算到 $2004$ 年年末，妈妈将津津平常存的钱加上 $20\%$ 还给津津之后，津津手中会有多少钱。

## 输入格式

$12$ 行数据，每行包含一个小于 $350$ 的非负整数，分别表示 $1$ 月到 $12$ 月津津的预算。

## 输出格式

一个整数。如果储蓄计划实施过程中出现某个月钱不够用的情况，输出 $-X$，$X$ 表示出现这种情况的第一个月；否则输出到 $2004$ 年年末津津手中会有多少钱。

注意，洛谷不需要进行文件输入输出，而是标准输入输出。

```c++
#include <bits/stdc++.h>
using namespace std;

int main() {
    int rem = 0, save = 0;
    for (int i = 1; i <= 12; ++i) {
        rem += 300;
        int cons; cin >> cons;
        rem -= cons;
        if (rem < 0) {
            cout << -i;
            return 0;;
        }
        if (rem >= 100) {
            int hund = rem / 100;
            rem -= 100 * hund;
            save += 100 * hund;
        }
    }
    cout << save * 0.2 + save + rem;
    return 0;
}
```

### 1. 收获一：精准的“状态机”建模（原理：时间顺序与变量同步）

这道题的本质是一个**离散时间状态机**。每个月初，系统状态（`rem`）必须严格按照“**先入账 → 再消费 → 后存款**”的顺序更新。

- **你的写法**：`rem += 300`（入账）→ `rem -= cons`（消费）→ `if(rem<0)`（判断）→ `if(rem>=100)`（存款）。
- **潜在坑点**：如果把“判断透支”放在“存款”之后，可能会把刚存进去的钱又取出来花，导致逻辑失真。你的顺序完美契合了现实生活的资金流转顺序，这是模拟题最重要的**时序思维**。

### 2. 收获二：整数运算对抗浮点精度（原理：数据类型与底层表示）

你最后写了 `cout << save * 0.2 + save + rem;`。虽然因为 `.0` 小数部分为 0，`cout` 默认会省略末尾的 `0` 和 `.`（如 `120.0` 输出为 `120`），所以**能通过**，但这里隐藏着两个重要的计算机原理：

- **浮点数误判风险**：`0.2` 在二进制中是无限循环小数（`0.00110011...`），浮点运算会产生极小的舍入误差（如 `0.0000001`）。虽然乘以 100 的倍数在这里恰好不会翻车，但在其他题目中，这可能导致输出 `119.999` 被判错。
- **强类型提升**：`save * 0.2` 将整个表达式隐式提升为 `double`，输出时涉及浮点流的格式化处理，效率低于整数输出。

**最佳实践（吸取的精华）**：永远在算法题中**优先使用整数除法**。因为 `save` 一定是 `100` 的倍数，利息 `save * 20 / 100` 完全等价于 `save / 5`。你应写成：

cpp

```
cout << rem + save + save / 5; // 纯整数运算，0精度损失，0性能开销
```



这教会你：**能用整数算，绝不用浮点**，这是竞赛选手的肌肉记忆。

### 3. 收获三：`return 0` 与 `break` 的底层差异（原理：控制流与栈帧）

你在循环内遇到透支时用了 `return 0;`（而不是 `break`），这是一个极佳的决策：

- **`break`**：仅跳出 `for` 循环，程序会继续执行循环外的 `cout`，如果此时 `rem<0`，你的 `cout` 依然会执行，可能输出负数年份后又输出了乱七八糟的存款数。
- **`return 0`**：**立即终止整个 `main` 函数**，销毁当前栈帧并直接返回操作系统，彻底杜绝了后续代码的执行。

这让你深刻理解了**“提前终止”**在异常处理中的威力——当业务逻辑走到“死路”时，果断“弃车保帅”，比设置层层嵌套的 `flag` 标志位要清晰得多。

### 4. 收获四：贪心策略的“无脑性”应用（原理：约束利用）

你的存款逻辑是 `int hund = rem / 100;`，直接取整百部分存入，剩余零头留着。

- **原理支撑**：题目保证“津津总是把整百的钱存起来”。你利用 C++ **整数除法向下取整**的特性，完美映射了这一规则，省去了 `if` 嵌套判断。
- **思维升华**：这教会你在处理约束条件时，**不要想复杂**。很多时候题目给的限制（如“整百”）本身就是给算法“开后门”，简化计算逻辑。读题时，要敏锐地捕捉这类“硬约束”并直接转化为代码的数学特性。
